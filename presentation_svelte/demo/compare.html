SVELTE 53 lignes
// Declarer un etat reactif let resources = $state([]); let filter = $state("all"); let newName = $state(""); // Modifier : DIRECT resources.push(newItem); // mutation OK filter = "rival"; // reassignment OK
REACT 188 lignes
// Declarer un etat reactif const [resources, setResources] = useState([]); const [filter, setFilter] = useState("all"); const [newName, setNewName] = useState(""); // Modifier : COPIE OBLIGATOIRE setResources(prev => // setter function [...prev, newItem]); // spread = copie setFilter("rival"); // setter aussi
Difference cle : React impose l'immutabilite — chaque modification cree une copie du state. Svelte permet la mutation directe. Le compilateur detecte les changements a l'avance. Resultat : moins de code, moins d'allocations memoire.
SVELTE 54 lignes
// Valeurs derivees — auto-recalculees let players = $derived( resources.filter(r => r.role !== "objet") ); let objects = $derived( resources.filter(r => r.role === "objet") ); // Pas de dependances a lister // Svelte les detecte automatiquement
REACT 188 lignes
// Valeurs derivees — deps manuelles const players = useMemo( () => resources.filter( r => r.role !== "objet"), [resources] // deps obligatoires ); const objects = useMemo( () => resources.filter( r => r.role === "objet"), [resources] // oubli = bug );
Piege classique React : Oublier une dependance dans le tableau de useMemo/useEffect = bug silencieux. Le linter peut aider mais pas toujours. Svelte detecte les dependances au moment de la compilation — impossible de se tromper.
SVELTE 55 lignes
// Countdown TTL chaque seconde setInterval(() => { resources.forEach(r => { if (r.ttl > 0) r.ttl--; }); }, 1000); // Mutation directe // Le compilateur met a jour // uniquement les noeuds TTL
REACT 1812 lignes
// Countdown TTL chaque seconde useEffect(() => { const interval = setInterval(() => { setResources(prev => prev.map(r => r.role === "objet" && r.ttl > 0 ? { ...r, ttl: r.ttl - 1 } : r ) ); }, 1000); return () => clearInterval(interval); }, []);
Immutabilite en action : React doit map sur tout le tableau et creer un nouvel objet {...r, ttl: r.ttl-1} pour chaque objet — meme ceux qui n'ont pas change. Svelte : r.ttl-- et le compilateur ne met a jour que le texte du TTL dans le DOM. C'est ce qu'on appelle l'update chirurgical.

bind:value en Svelte remplace le pattern value={x} + onChange={e => setX(e.target.value)} de React. Une directive au lieu de deux props. Et pas besoin de useCallback pour eviter les re-renders — Svelte ne re-render pas les composants comme React.

Pourquoi 0 KB ? Svelte est un compilateur. Au build, il transforme vos composants en JS vanilla optimise. Il n'y a pas de fichier "svelte.js" envoye au navigateur. React, lui, doit envoyer son runtime (react + react-dom) pour que l'app fonctionne. Verifiable dans l'onglet Network de Chrome DevTools.

Svelte gagne sur

+ Taille du bundle (0 KB runtime)
+ Quantite de code (-40%)
+ Simplicite (0 hooks a apprendre)
+ Performance (update chirurgical)
+ Animations natives
+ CSS scope natif
+ Courbe d'apprentissage
+ Pas d'immutabilite requise

React gagne sur

+ Ecosysteme (225K stars, milliers de libs)
+ Marche de l'emploi (beaucoup + d'offres)
+ Communaute (Stack Overflow, tutos)
+ Entreprises (Meta, Netflix, Airbnb)
+ React Native (mobile natif)
+ Maturite en production a grande echelle
~ TypeScript (support equivalent)
~ Outils dev (React DevTools mature)

Conclusion

Svelte est superieur techniquement : moins de code, plus performant, plus simple. React domine par son ecosysteme et son adoption. Pour un projet comme Mission Nautilus (jeu mobile geolocalise, performance critique), Svelte est le meilleur choix. Pour une app enterprise avec 50 developpeurs, React reste plus prudent.
